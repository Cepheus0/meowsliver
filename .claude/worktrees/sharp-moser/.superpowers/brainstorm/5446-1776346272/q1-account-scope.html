Q1 — Account concept ทำงานยังไงกับ transaction เดิม?

นี่คือ fundamental decision ที่กระทบ schema, import, และ UX ทั้งหมด

ตอนนี้ flow เป็นแบบนี้

Meowjot CSV → Transaction (income/expense/transfer) → Analytics

Transaction ไม่รู้ว่า "เงินอยู่ในกระเป๋าไหน"

A

Linked — Transaction ผูก account

แต่ละ transaction มี accountId (income → credited to, expense → debited from). Balance คำนวณจาก transaction จริง.

ข้อดี

  • Balance ถูกต้องเสมอ (derived from transactions)
  • Track เงินได้ว่าไหลจากไหนไปไหน
  • ตรงกับ CoinKeeper concept จริงๆ

ข้อเสีย

  • ต้อง migration: transaction เดิม 771 รายการต้อง assign account
  • Import wizard ต้องเพิ่ม step เลือก account
  • ซับซ้อนกว่า Option B
B

Parallel — Account เป็น snapshot อิสระ

Account เก็บ balance ด้วยตัวเอง (manual entry หรือ bank statement sync) ไม่ต้องผูกกับ transaction เดิม. Transaction ยังคงเป็น flow analytics เหมือนเดิม.

ข้อดี

  • ไม่กระทบ transaction เดิมเลย — zero migration
  • ง่ายกว่า implement มาก
  • Balance อัปเดตได้ง่าย (แค่ใส่ตัวเลขล่าสุด)

ข้อเสีย

  • Balance อาจไม่ sync กับ transactions (double truth)
  • ไม่รู้จริงๆ ว่า "transaction ไหนเข้าบัญชีไหน"
  • Net worth อาจนับซ้ำ (transactions + account balance)
C

Hybrid — Account snapshot + soft link

Account มี balance ของตัวเอง (snapshot) แต่ transaction มี optional accountId. ไม่ enforce, ไม่ compute balance from transactions — แต่ช่วย filter ได้.

ข้อดี

  • Migration ง่าย (accountId ใส่หรือไม่ใส่ก็ได้)
  • User เลือก granularity เองได้
  • Upgrade path: ทำ Option A ได้ในอนาคต

ข้อเสีย

  • Balance ยัง "manual" อยู่ ไม่ self-consistent
  • Design ซับซ้อนกว่า B เล็กน้อย

💡 คำแนะนำของผม: Option C (Hybrid) — เริ่มง่าย ไม่ break ของเดิม แต่มี upgrade path ไป Option A เมื่อ user พร้อม
